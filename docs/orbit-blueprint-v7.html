01 ──

The 7 Foundational Principles

Every decision in this blueprint follows these 7 laws. If a new idea breaks a principle, it doesn't go in. Yeh architecture ki constitution hai.

PRINCIPLE 01
Serverless by Default
Koi khud ka server nahi. Vercel Edge + Firebase + R2 = auto-scale. Kabhi bhi DevOps hire nahi karna padega Phase 3 tak.
PRINCIPLE 02
Managed First, Self-Host Later
Pusher, Agora, Gemini — sab managed. Jab scale aa jaaye (1M+ MAU) tab migrate specific hot paths. Never premature optimization.
PRINCIPLE 03
Cost = f(Revenue)
Infra cost user count ke saath automatic scale hota hai. Subscription upgrade karo jab revenue milne lage, pehle nahi.
PRINCIPLE 04
Denormalize for Reads
Firestore reads cheap hain, joins nahi. Har feed/profile/room me counter aur cached fields pre-compute karo. Read = 1 doc.
PRINCIPLE 05
Ship, Measure, Iterate
PostHog har event track karta hai Day 1 se. Bina analytics ke ek bhi feature ship nahi karna. Blind flying = death.
PRINCIPLE 06
Compliance from Day 0
DPDP 2023 aur IT Rules 2021 baked-in. Delete account, data export, consent — first build hi mein, baad mein add karna 10x mehenga.
PRINCIPLE 07
One Codebase, All Platforms
Next.js 15 = web + PWA. Capacitor = native iOS/Android wrapper (push + deep links + Play Store). Zero duplicate code.
💡 Why these 7 matter: Kayi apps Phase 1 mein ship kar dete hain but Phase 3 mein rewrite karte hain (complete code dobara) kyunki foundation kamzor thi. Yeh 7 principles rewrite rokte hain. Agar kal ko koi naya feature "lagta hai cheap hai lekin principle break kar raha hai" — toh mat karo, baad mein dard hoga.

Database Schema at 10M Scale

Firestore ka dark side: 1 write/sec per doc limit, no joins, reads = money. Denormalize or die. Yeh schema pre-computed counters + cached fields ka use karti hai to hit 10M without rewriting.

⚠️ Firestore Limits Jo Beginners Miss Karte Hain: → 1 write/sec per document (hot counter problem)
→ 1 MB max document size
→ 100 levels deep max nesting
→ No full-text search (need Meilisearch — see Section 04)
→ No SQL JOINs (denormalize everything)
→ Reads charged per document, not per query

Denormalization Strategy
Why author data is copied
Har message mein author.username aur author.avatar copied hote hain. Kyun? Kyunki agar sirf authorUid store karein toh feed render karne ke liye 100 messages = 100 user doc reads = 100x cost. Denormalization mein "stale data" ka problem hota hai — but ORBIT mein usernames rarely change, aur avatar change pe we do batch update via Cloud Function.
Sharded Counters
Hot Room Problem Solve
Agar ek room mein 1000 log ek saath message karte hain, sab memberCount++ karna chahte hain = 1 write/sec limit fail. Solution: 10 shards banao — har write random shard pe ho, read time pe sum kar do. _counters/roomX/shards/s1-s10 = 10 writes/sec safe.
Required Composite Indexes
firestore.indexes.json
rooms: [language, activeUntil DESC] rooms: [type, memberCount DESC] messages: [author.uid, createdAt DESC] bazaar: [category, createdAt DESC] bazaar: [language, priceINR ASC] reports: [status, createdAt DESC] challenges: [category, endsAt DESC]
TTL + Cleanup
Auto-delete old data
Messages older than 1 year auto-archive to R2 (via nightly Cloud Function)
Inactive mood rooms (activeUntil < now) mark archived: true
Ended challenges archived after 30 days
Deleted user data purged 30 days after deletion (DPDP req)

Moat & Positioning

Investors yeh question karte hain: "Meta/Google tomorrow clone kar de, tab tum kya karoge?" Jawab: ORBIT ka moat 4 layers ka hai.

Moat Layer 1
🏘 Community Density
Network effects compound. Har user naye user ke liye value add karta hai. Mood Rooms mein jitne log, utna better matching. Meta ka WhatsApp India has 500M users — but WhatsApp mein rooms/moods/skill-bazaar nahi hai.
ORBIT = social graph + interest graph + commerce graph. Triple overlay = replication hard even for Meta.
Moat Layer 2
💰 Economic Gravity
Skill Bazaar sellers apni reputation, reviews, customer base build karte hain. Switching cost HIGH. Seller apna ₹50K/mo income nahi chodega new platform ke liye.
Credits + Karma + cashout ledger = users financially invested. Leave = lose money. This is the moat that WhatsApp doesn't have.
Moat Layer 3
🆔 Identity Portability
ORBIT Card = user ka digital resume + portfolio. Shared link on LinkedIn, Instagram, WhatsApp bio. Backlinks to ORBIT profile.
Identity stickiness: users apni ORBIT ID badalne mein hesitate karenge. Similar to how Gmail ID you never change.
Moat Layer 4
🌐 Cultural Depth
Hinglish smart replies, regional rooms, Indian festivals theming, UPI-native cashout. Meta ke USA-first product designers yeh subtlety nahi samajhte.
Cultural moat = jitni deep, utni strong. ORBIT ko build kar raha hai ek Indian who understands Chandigarh + Mumbai + Chennai.
Positioning Statement (1 Sentence)
"ORBIT is WhatsApp's social warmth × LinkedIn's economic graph × Discord's community rooms × Instagram's identity polish — built specifically for 1.4 billion Indians who speak 22 languages."
